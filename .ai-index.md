## 📋 Framework Boundaries (MUST READ)

### Core Architecture

```
SDD Framework (v3.0 Final)
│
├── CORE (4 immutable mandates)
│   └── Cannot be modified
│
├── CLIENT (151 customizable guidelines)
│   └── Can be selected/customized per project
│
└── GOVERNANCE MODEL
    ├── SHA-256 fingerprinting (integrity)
    ├── Salt strategy (tampering detection)
    └── Language-based customization (Python/Java/JavaScript)
```

### Your Rules (Non-Negotiable)

File: [.sdd-core/HONEST-CRITIQUE-CONSTITUTION.md](./.sdd-core/HONEST-CRITIQUE-CONSTITUTION.md)

**Sample rules you MUST follow:**
- ✅ All features need tests (100% coverage requirement)
- ✅ Tests must pass before marking done
- ✅ Document decisions in checkpoints
- ✅ Follow ports & adapters pattern (ADR-003)
- ✅ Respect thread isolation (check execution-state/ first)
- ✅ Use "Definition of Done" checklist (45+ criteria)

**What happens if you break rules:**
- 🚫 PR cannot be merged
- 🚫 Tests won't pass  
- 🚫 Architecture validation fails
- → Checkpoint why, ask for guidance

### Your Boundaries

- ✅ **CAN:** Modify code in `.sdd-core/`
- ✅ **CAN:** Modify code in project directories
- ✅ **CAN:** Modify CANONICAL/decisions/ (document why)
- ✅ **CAN:** Create custom/[project-name]/ configurations
- ❌ **CANNOT:** Modify HONEST-CRITIQUE-CONSTITUTION.md unilaterally
- ❌ **CANNOT:** Change immutable CORE mandates
- ❌ **CANNOT:** Create new test frameworks (use pytest)
- ❌ **CANNOT:** Bypass rule enforcement

## 7️⃣ The Workflow (Your 7-Phase Process)

Every feature/bug follows this exact workflow. Non-negotiable:

```
PHASE 0: Setup infrastructure (.ai/ context)
   ↓ (20-30 min, one time only)

PHASE 1: Lock to Rules (constitution + rules)
   ↓ (15 min per task)

PHASE 2: Check Execution State (detect conflicts)
   ↓ (5 min)

PHASE 3: Choose PATH (bug=A, simple=B, complex=C, multi=D)
   ↓ (5 min)

PHASE 4: Load Context (search .ai/runtime/)
   ↓ (10-20 min)

PHASE 5: Implement (code + tests, TDD)
   ↓ (1-8 hours depending on complexity)

PHASE 6: Validate (tests + definition-of-done)
   ↓ (10-15 min)

PHASE 7: Checkpoint (document + PR)
   ↓ (10 min)

✅ DONE: Feature complete, ready for review
```

**Complete details:** [.sdd-core/spec/guides/onboarding/AGENT_HARNESS.md](./.sdd-core/spec/guides/onboarding/AGENT_HARNESS.md)

## 🐛 Bug Fix Workflow

### Your Task: Fix a bug

```bash
# Step 1: Understand the bug
git log --oneline -20                    # Recent changes
grep -r "error_pattern" tests/           # Find test case

# Step 2: Run tests to confirm failure
python3 run-all-tests.py --fail-fast      # See which test fails

# Step 3: Locate the code
# Most bugs are in: .sdd-core/, .sdd-wizard/, .sdd-compiler/

# Step 4: Write failing test first (TDD)
pytest tests/test_bug_fix.py -v           # Should fail

# Step 5: Fix the code
# Make it pass the test

# Step 6: Run all tests
python3 run-all-tests.py                  # All must pass

# Step 7: Checkpoint
echo "FIXED: [bug description]" > CHECKPOINT.md
```

**Example bug checkpoint:**
```markdown
# Checkpoint: Phase 4 NameError Fix

**WHAT:** Fixed NameError in phase_4_filter_guidelines.py
**WHY:** Profile parameter removed but still referenced  
**HOW:** Removed line with 'profile': profile from report dict
**TESTS:** 16/16 Phase 4 tests now passing  
**RISK:** Low - isolated to one file
**STATUS:** Ready for review
```

## ✨ New Feature Workflow

### Your Task: Implement a feature

```bash
# Step 1: Understand requirements
# Read: Issue, requirements.md, related decisions

# Step 2: Check if someone else is working on it
cat .ai/execution-state/_current.md       # See who's working

# Step 3: Update execution state (claim the task)
echo "Agent: [your-name]
Task: [feature-name]
Start: $(date)" > .ai/execution-state/_current.md

# Step 4: Design (Architecture Decision Record)
# Write: Why this design? Alternatives?
vi .sdd-core/spec/CANONICAL/decisions/ADR-007-your-feature.md

# Step 5: Write tests FIRST (TDD)
# Create: tests/test_your_feature.py
# Make sure they fail initially

# Step 6: Implement code
# Make tests pass
pytest tests/test_your_feature.py -v

# Step 7: Run all tests
python3 run-all-tests.py --verbose

# Step 8: Check definition of done
# Verify all 45+ criteria
vi .sdd-core/spec/CANONICAL/specifications/definition-of-done.md

# Step 9: Checkpoint
echo "IMPLEMENTED: [feature]" > CHECKPOINT.md
```

**Example feature checkpoint:**
```markdown
# Checkpoint: New CLI Command

**WHAT:** Added `sdd governance validate` command

**HOW:** 
- Extended Typer CLI with new command

- Returns SHA-256 fingerprint verification results

- 5 new unit tests (validation success/failure cases)

**RISK:** Medium - touches CLI interface
**STATUS:** Ready for PR review
```

## 🚨 Emergency / Troubleshooting

### Tests Failing?

```bash
# 1. Run with verbose to see what failed
python3 run-all-tests.py --verbose

# 2. Run specific failing test
pytest tests/test_file.py::TestClass::test_method -v

# 3. Check if it's expected (profile-related)
grep -r "profile" .sdd-wizard/orchestration/
# Result: Profiles removed in v3.0 by design

# 4. Not sure what to do?
cat .sdd-core/spec/guides/emergency/when-tests-fail.md
```

### Rule Violation?

```bash
# 1. Check what rule was broken
cat .sdd-core/HONEST-CRITIQUE-CONSTITUTION.md | grep "your_rule_number"

# 2. See examples of doing it right
grep -r "example_correct_pattern" .sdd-core/spec/CANONICAL/specifications/

# 3. Get emergency help
cat .sdd-core/spec/guides/emergency/when-rules-violated.md
```

### Confused About Architecture?

```bash
# 1. Read the decision
cat .sdd-core/spec/CANONICAL/decisions/ADR-003.md  # Ports & adapters

# 2. See implementation example
find . -path "*adapters*" -name "*.py" | head -5 | xargs head -20

echo "QUESTION: Why ports & adapters?" > CHECKPOINT.md
```

## ❓ FAQ

### Q: Where do I see what tests exist?
```bash
python3 run-all-tests.py --list-layers
# Output: Lists all 7 layers with test counts
```

### Q: Can I run just the tests I care about?
```bash
python3 run-all-tests.py --layer "Wizard"  # Only wizard tests
pytest tests/test_specific.py               # Single test file
```

### Q: How do I know if my code is done?
```bash
# Check against 45+ criteria:
cat .sdd-core/spec/CANONICAL/specifications/definition-of-done.md

# Run full test suite:
python3 run-all-tests.py
```

### Q: What's the "Definition of Done"?
A file with 45+ criteria your code must meet. Read it before marking anything "done".

### Q: Can I modify the CONSTITUTION?
A: No. It's immutable. If you think it needs changing, ask first (checkpoint).

### Q: Who else is working right now?
A: Check `.ai/execution-state/_current.md` (see if someone claimed the task)

### Q: Can I work on multiple tasks?
A: No. Thread isolation = one task at a time. Finish → Checkpoint → Move to next.

### Q: How do I report what I did?
A: Create a CHECKPOINT file describing:
- WHAT: What did you implement?
- WHY: Why this approach?
- HOW: How did you implement it?
- TESTS: What tests pass?
- RISK: What could go wrong?
- STATUS: Ready for review?
